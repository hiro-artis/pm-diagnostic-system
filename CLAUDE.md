# PM協会テスト診断システム - 実装計画

## プロジェクト概要

**目的：** PM適性診断を自動化するAIエージェントシステムの構築

**診断対象：** PM候補者・既存PM

**判定フロー：**
1. **1次テスト（筆記）** → PM基礎知識 + オフィス移転実務 + **マインドセット（最重要）**
2. **2次テスト（面接）** → マインドセット深掘り検証（1次で60点以上の場合のみ実施）
3. **最終評価** → A/B/C/再検査の4段階判定

**マインドセット基準：** 未来志向・自責・優しさ・聴く力・置いてきぼりにしない・一人で進まない

## 主要な実装ポイント

### 1. マインドセット採点の精度化

**課題：** AIによる採点にばらつきが出ないようにする

**対策：**
- 各シナリオの選択肢別にマインドセット評価パターンをハード定義
- Claude APIに「評価パターン」を明示指示
- 複数回の評価で平均化
- 信頼度スコア（confidence score）を併用

**実装例：**
```python
# マインドセット評価パターン定義
MINDSET_PATTERNS = {
    "MS-S01": {
        "A": {"futureFocused": 3, "selfResponsibility": 2, "confidence": 0.8},
        "B": {"futureFocused": 2, "selfResponsibility": 4, "confidence": 0.85},
        "C": {"futureFocused": 5, "selfResponsibility": 5, "confidence": 0.95},  # 正答
        "D": {"futureFocused": 1, "selfResponsibility": 1, "confidence": 0.9}
    }
}
```

### 2. 1次テストと2次テストの一貫性確認

**課題：** 1次で高スコアでも、2次で矛盾した回答をする可能性

**対策：**
- 2次面接で「1次テストのシナリオXでYを選んだ理由」を具体的に質問
- 回答の一貫性を採点時に評価
- 矛盾が大きい場合は「再検査」に振る

### 3. 記述式採点の標準化

**課題：** AI採点の結果が評者によってばらつく可能性

**対策：**
- 評価ディメンション（リスク認識、実現性、コミュニケーション等）を明確に定義
- 各ディメンション別の評価ルーブリックを作成
- 複数の採点パターン（厳格・標準・緩和）を用意
- 実装段階で「複数エージェント採点 → 平均」方式を検討

### 4. ユーザー体験の向上

**課題：** テストが長い（約90分）とドロップアウトの可能性

**対策：**
- 適宜進捗表示・励ましメッセージ
- 中断機能（ここまでで一旦終了→後日再開）
- モバイル対応の検討
- 各セクション完了時に小さなご褒美表示（スコア先出しなど）

## 技術選定とAPIの使用

### Claude API活用

**エージェント別API活用パターン：**

| エージェント | 用途 | 呼び出し回数 |
|---|---|---|
| 基礎知識テスト | 問題配信（定型テンプレート）/ 確信度チェック（オプション） | 最小限（1-2回） |
| オフィス移転テスト | 記述式採点 | 3回（各問題） |
| マインドセット | シナリオ配信（定型）/ 選択肢評価（オプション） | 最小限 |
| マインドセット面接 | 質問生成・回答分析・スコア計算 | 1セッション=5-8回 |
| 詳細採点 | 複雑な採点・再評価 | 必要時のみ |

**コスト削減：**
- Prompt Caching活用（テスト問題セット・評価基準をキャッシュ）
- バッチAPI検討（結果処理がバッチ可能な場合）

### エラーハンドリング

**主要なエラーケース：**
1. API接続失敗 → リトライ（3回）+ タイムアウト（30秒）
2. テスト中断 → セッション保存 → 後日再開
3. 採点エージェント遮断 → 管理者通知 + 仮スコア表示
4. ユーザー入力エラー → 再入力促す + 最大3回まで

## 注記・制約条件

### 重要な設計決定

1. **マインドセットを最優先** → 基礎知識・オフィス移転の採点よりマインドセットの精度を重視
2. **2次テスト実施率** → 1次テストの70%が2次進出（マインドセット60点以上）と予想
3. **再検査フロー** → 1次テストでいずれかのセクション不合格 → 1週間後に再受検可能

### 前提条件

- Claude API（or Anthropic SDK）の利用可能性を確認
- ベータテストに協力可能な人数の確保（5-10名）
- 実装者のPython3.9以上の環境

### リスク

| リスク | 対策 |
|-------|------|
| AI採点のばらつき | 評価パターン定義 + 複数エージェント採点 |
| 実装スケジュール遅延 | フェーズごとの MVP 定義 + 優先順位の厳格化 |
| ベータテスター確保困難 | 早期に協力者との関係構築 |
| API コスト超過 | Prompt Caching 活用 + 呼び出し最小化設計 |
